This project handles game data safely via local JSON files and an optional online sync mechanism. All runtime settings are managed through the `launch.py` wizard and saved to `config/settings.json`.

## Configuration Files

<CardGrid>

<Card title="config/settings.json" icon="setting">
  Stores ADB host/port, wait multipliers, sync toggle, and target platform.
  <br />
  <strong>
    Managed automatically by <code>launch.py</code>.
  </strong>
</Card>

<Card title="config/screen_config.json" icon="window">
  Lists enabled screens, grid coordinates, and menu navigation rules.
</Card>

</CardGrid>

<Aside type="danger">
  **Do not edit `config.py` directly.** It acts as a loader/model validator for
  `settings.json`. Manual edits will be overwritten on the next launch.
</Aside>

## What Data Files Matter?

The processors read these local files:

- `assets/data/equipment_processed.json` - equipment definitions used for matching
- `assets/data/items_processed.json` - item definitions
- `assets/data/students_processed.json` - student metadata

If a file is missing, the app falls back to defaults. Always back up before manually replacing them.

## Data Sync

By default, the app uses local data. To enable online updates:

<Steps>

1. Run `python launch.py` and answer **Yes** to `Enable online data sync?`.

2. On startup, the scanner will attempt to download updated JSONs from the configured GitHub repo.

3. Files are compared via SHA256 and written atomically (`*.tmp` → rename). If the network fails, your local files remain untouched.

</Steps>

### Manual Sync

Use the standalone CLI to run sync without starting the main app:

```bash
python -m tools.sync_data
```

## Performance Tuning (Wait Multipliers)

If your device is slow or navigation fails, adjust these in `config/settings.json` or via `python launch.py -e`:

<CardGrid>

  <Card title="wait_multiplier" icon="clock">
    Default: <code>1.0</code><br />
    Scales short action delays like taps and small waits between captures.
  </Card>

  <Card title="wait_screen_nav_multiplier" icon="rocket">
    Default: <code>2.0</code><br />
    Scales navigation delays such as screen loads and menu transitions.
  </Card>

</CardGrid>

<Aside type="tip">
  <strong>Recommended for laggy devices:</strong>
  <br />
  <code>wait_multiplier: 1.2</code>
  <br />
  <code>wait_screen_nav_multiplier: 3.0</code>
</Aside>

## Troubleshooting

<CardGrid>

  <Card title="Sync failed" icon="cloud-download">
    Check your internet connection. The manager retries 3 times with exponential backoff.<br /><br />
    To debug manually:
    ```bash
    python -m tools.sync_data
    ```
  </Card>

  <Card title="Screens not detected" icon="list-format">
    Increase <code>wait_screen_nav_multiplier</code>.<br />
    Ensure <code>screen_config.json</code> has <code>enabled: true</code> for the required screens.
  </Card>

  <Card title="Linux / AMD GPU screenshot corruption" icon="warning">
    ADB <code>exec-out</code> may sometimes inject stderr into PNG output.<br /><br />
    The tool already sanitizes this, but updating ADB is recommended.
  </Card>

</CardGrid>

## Advanced Notes

Files are written atomically (tmp file then rename) to avoid partial writes.

The ADB screenshot bug on some Linux setups (AMD GPU) can corrupt PNG bytes. If screenshots fail to load, ensure `exec-out` uses stderr redirection:

```bash
adb -s <serial> exec-out 'screencap -p 2>/dev/null'
```

Or update `utils/device/adb_controller.py` with the same capture command.
